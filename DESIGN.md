## Problem Statement

Build a neal.fun-inspired game site — a growing collection of polished two-player browser games. Starting with Connect Four that can be played in real-time between two people on separate devices. The core value proposition is the FEEL: satisfying micro-interactions, spring physics, playful aesthetic. Not a game engine — a sensory experience that happens to be multiplayer.

## What Makes This Cool

The neal.fun aesthetic applied to multiplayer. Every piece drop has weight and bounce. Winning feels celebratory. The site itself feels like a handmade thing between two people, not a generic board game app. The "apartment" concept (from cross-model review) — a shared space where the games live — could make this feel like YOUR place rather than just another game site.

## Constraints

- Two players only (for now — just the builder and their girlfriend)
- Must work on mobile (she'll open the link on her phone)
- No auth for v1 — link sharing IS the access model
- Supabase free tier for backend (generous enough for personal use)
- Vercel free tier for hosting
- Must feel polished even in v1 — "delight-first" means the first shipped version should make you smile

## Premises

1. **Link-based access** — create a game, get a shareable URL. The game creator is always player 1; whoever opens the shared link is player 2. Player identity is a random UUID stored in localStorage and written to the game row (`player1_id` / `player2_id`). On reconnect, match localStorage UUID to the game row to determine your role. Additional visitors see a "game full" message. No auth needed for v1.
2. **Supabase Realtime Postgres Changes** for live game sync — both players subscribe to changes on their game row via the game ID. A single DB write both persists the move and triggers the notification. No game server, no anti-cheat (you trust your opponent). No server-side move validation in v1.
3. **Game state in Postgres** — one row per game (board state as JSON array, whose turn, winner status). Source of truth on refresh.
4. **Delight over architecture** — invest in feel (spring animations, sounds, confetti) before investing in multi-game infrastructure. Multi-game structure comes when game #2 arrives.
5. **neal.fun aesthetic** — light/warm off-white backgrounds, playful sans-serif typography (Satoshi or General Sans), generous spacing, spring animations, no visual clutter, rounded corners everywhere.

## Approaches Considered

### Approach A: Ship Tonight (Minimal Viable)
Single Connect Four page at `/connect-four/[gameId]`. No homepage, no game selection. Game state in Supabase, Realtime sync. Basic animations. Playing in hours.
- Pros: Fastest path to playing
- Cons: No site wrapper, bare-minimum feel, adding games later requires structural work

### Approach B: Multi-Game Foundation
Full site structure from day one: homepage, game list, shared lobby component, localStorage player names, game framework for adding new games.
- Pros: Future-proof, proper architecture
- Cons: More upfront work before actually playing, potential over-engineering

### Approach C: Delight-First (CHOSEN)
Ship a single polished Connect Four game fast, but invest in FEEL: spring physics piece drops, satisfying sounds, confetti on win. Architecture for multi-game comes later. Emotional experience first.
- Pros: Playing tonight AND it feels incredible, neal.fun magic is in micro-interactions not architecture
- Cons: Multi-game structure deferred, sound/animation polish can be a rabbit hole

## Recommended Approach

**Approach C: Delight-First.** The whole point of this project is that it feels amazing. Architecture is easy to add later; delight needs to be baked in from the start.

### Database Schema
```sql
create table games (
  id uuid primary key default gen_random_uuid(),
  game_type text not null default 'connect-four',
  -- Board is column-major: 7 arrays (one per column), each containing up to 6 values (1 or 2) bottom-to-top
  -- A move in column N pushes the player number (1 or 2) to the end of board[N]. Array length = stack height (max 6).
  board jsonb not null default '[[],[],[],[],[],[],[]]',
  -- current_turn: 1 = player 1's turn, 2 = player 2's turn
  current_turn int not null default 1,
  winner int,
  player1_id uuid,
  player2_id uuid,
  player1_name text,
  player2_name text,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

### Real-time Flow

**Mechanism:** Supabase Realtime Postgres Changes (listen to updates on the `games` table). A single DB write both persists the move AND triggers the notification to the opponent. Simpler than Broadcast for this use case since the DB is already the source of truth.

1. Player 1 clicks "New Game" → creates a row in `games`, gets URL `/connect-four/[gameId]`
2. Player 1 shares URL with Player 2
3. Both players subscribe to Supabase Realtime Postgres Changes filtered to their game ID
4. On move: update the board JSON in Postgres → Realtime pushes the change to the opponent → opponent's board animates the piece drop
5. Win detection runs client-side after each move; the player who makes the winning move writes the winner field to the DB
6. On page refresh: read the current game state from Postgres (source of truth)

**Optimistic UI:** The dropper sees their piece animate immediately (optimistic). When the Postgres Changes event arrives back to the dropper, it is suppressed (compare incoming board state to local state — if identical, ignore the echo).

5. **Deploy:** Ship to Vercel. Send the link.

### Key Micro-Interactions
- **Piece drop:** Framer Motion spring (stiffness ~200, damping ~12). Piece accelerates down, bounces slightly on landing. Subtle screen shake on impact.
- **Hover preview:** Ghost piece appears at top of column on hover (opacity 0.3), disappears when cursor leaves.
- **Turn indicator:** Smooth color transition showing whose turn it is.
- **Win celebration:** 4 winning pieces pulse/glow, then canvas-confetti burst from center. Satisfying "victory" sound.
- **Draw state:** Pieces slowly desaturate, playful "oh well" copy appears.

## Open Questions

1. **Sound library choice:** use-sound (simpler, React hook) vs Howler.js (more control, sprite sheets for multiple sounds)? Leaning use-sound for v1 simplicity.
2. **The "apartment" concept:** Love it as a future evolution. Should it be the v2 homepage, or is a simple game list better? Deferring this decision until after Connect Four ships.
3. **Player names:** localStorage is fine for v1, but what happens when she clears her browser data? Accept the limitation for now.
4. **Presence:** Show a green dot when opponent is in the game? Supabase Presence supports this. Nice-to-have for v1.
